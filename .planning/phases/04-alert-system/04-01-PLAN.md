    Create `014_alert_trigger_and_rls.sql` wrapped in `BEGIN; ... COMMIT;`.

    1. Drop the old trigger and function: `DROP TRIGGER IF EXISTS trigger_generate_alert ON telemetry_logs;` then `DROP FUNCTION IF EXISTS generate_alert_if_needed();`.

    2. Create `CREATE OR REPLACE FUNCTION check_telemetry_alerts() RETURNS TRIGGER` as `plpgsql` `SECURITY DEFINER`. Logic:
       - `SELECT fleet_id INTO v_fleet_id FROM vehicles WHERE id = NEW.vehicle_id;` — `RETURN NEW` if NULL.
       - Battery: if `NEW.voltage < 11.5` insert `BATTERY_CRITICAL` (severity `critical`); else if `NEW.voltage < 11.8` insert `BATTERY_WARNING` (severity `warning`). Hardcoded thresholds per D-04.
       - Coolant: if `NEW.temp > 110` insert `COOLANT_CRITICAL` (severity `critical`); else if `NEW.temp > 100` insert `COOLANT_WARNING` (severity `warning`). Per D-04.
       - DTC: if `NEW.dtc_codes` is non-empty, `FOREACH dtc_code IN ARRAY NEW.dtc_codes` insert one row per code, `code = dtc_code`, severity `warning` (all DTC alerts are `warning` from the trigger per RESEARCH Pitfall 5 / assumption A5), `message = 'DTC fault code: ' || dtc_code`, `dtc_codes = ARRAY[dtc_code]`. Put the FOREACH inside its own `DECLARE dtc_code TEXT; BEGIN ... END;` sub-block.
       - Every INSERT must be guarded by a dedup `NOT EXISTS` check per D-01/D-08: skip if an alert with the same `vehicle_id` AND same `code` (for battery/coolant use the warning OR critical code pair so a warning does not fire while a critical is active) AND `acknowledged = false` AND `created_at > NOW() - INTERVAL '1 hour'` already exists.
       - Each INSERT sets `vehicle_id`, `fleet_id` (= v_fleet_id), `severity`, `code`, `message`, `lat` (= NEW.lat), `lng` (= NEW.lng); DTC inserts also set `dtc_codes`.
       - `RETURN NEW;` at the end.

    3. `CREATE TRIGGER check_telemetry_alerts_trigger AFTER INSERT ON telemetry_logs FOR EACH ROW EXECUTE FUNCTION check_telemetry_alerts();`

    4. RLS on `alerts`: create `CREATE POLICY "Users can read own fleet alerts" ON alerts FOR SELECT USING (...)` — fleet members via `fleet_members.user_id = (auth.jwt()->>'sub')` OR org owners via `organizations.owner_id = (auth.jwt()->>'sub')` joined through `fleets`. Create `CREATE POLICY "Users can acknowledge own fleet alerts" ON alerts FOR UPDATE USING (<same membership/owner check>) WITH CHECK (acknowledged = true)`. Use `auth.jwt()->>'sub'` — NEVER `auth.uid()`. Drop any conflicting same-named policies first with `DROP POLICY IF EXISTS`.

    5. `ALTER PUBLICATION supabase_realtime ADD TABLE alerts;` — wrap defensively (if already a member this errors; use a `DO $$ ... EXCEPTION WHEN duplicate_object THEN NULL; END $$;` block or check `pg_publication_tables` first).

    6. Optionally `DROP FUNCTION IF EXISTS acknowledge_alert(UUID, UUID);` — the stale RPC has a UUID id mismatch (alerts.id is BIGSERIAL). Drop it to prevent misuse.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| parser/service-role → telemetry_logs | Untrusted device-derived telemetry crosses into the DB; trigger runs on every insert |
| browser client → alerts table | Authenticated user reads/acknowledges; must be scoped to their fleet |
| trigger function → vehicles/alerts | Trigger runs as SECURITY DEFINER (postgres role) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Information Disclosure | alerts SELECT — cross-fleet read | mitigate | RLS SELECT policy joins `fleet_members` / `organizations` on `auth.jwt()->>'sub'`; non-members get 0 rows (Task 2) |
| T-04-02 | Tampering | alerts UPDATE — acknowledge another fleet's alert | mitigate | RLS UPDATE policy same membership/owner check (Task 2) |
| T-04-03 | Tampering | alerts UPDATE — revert acknowledged=true back to false | mitigate | UPDATE policy `WITH CHECK (acknowledged = true)` — can only set true (Task 2) |
| T-04-04 | Denial of Service | Alert spam via rapid telemetry inserts | mitigate | 1-hour dedup `NOT EXISTS` guard per vehicle+code in trigger (Task 2, D-01/D-08) |
| T-04-05 | Elevation of Privilege | Trigger reads vehicles / inserts alerts without a user session | accept | Intentional: `SECURITY DEFINER` required so detection works server-side with no logged-in user; function body is fixed SQL with no dynamic input execution — injection surface is none |
| T-04-06 | Injection | DTC code string used in trigger message/insert | mitigate | `dtc_code` is bound as a value in parameterized `INSERT ... VALUES`, never concatenated into executed SQL; `code` column is `VARCHAR(50)` (Task 2) |
</threat_model>
