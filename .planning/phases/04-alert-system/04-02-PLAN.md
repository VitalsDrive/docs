<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser client → Supabase alerts table | Authenticated user SELECTs and UPDATEs alert rows; must stay fleet-scoped |
| Supabase Realtime → browser | INSERT events broadcast to subscribed dashboards; client must filter to its own fleets |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-07 | Information Disclosure | Realtime subscription delivers alerts for other fleets | mitigate | `subscribeToAlerts` handler filters `fleetIds.includes(alert.fleet_id)` before adding to state (Task 1); server-side RLS on the table is the backstop (plan 01 T-04-01) |
| T-04-08 | Information Disclosure | alertResource SELECT returns cross-fleet rows | mitigate | Query scoped `.in('fleet_id', fleetIds)`; RLS SELECT policy (plan 01) enforces server-side even if client query is tampered |
| T-04-09 | Tampering | Client acknowledges an alert outside its fleet | mitigate | RLS UPDATE policy (plan 01 T-04-02) blocks server-side; client `acknowledgeAlert` only sends `acknowledged=true` satisfying `WITH CHECK` |
| T-04-10 | Tampering | Client attempts to revert acknowledged=false | mitigate | `acknowledgeAlert` only ever writes `acknowledged: true`; RLS `WITH CHECK (acknowledged = true)` rejects false (plan 01 T-04-03) |
| T-04-11 | Spoofing | Forged Realtime payload injects a fake alert into UI | accept | Realtime payloads originate from the Supabase service; an attacker with channel access cannot inject INSERT events without DB write access (gated by RLS); UI-only display, no privileged action taken on payload |
</threat_model>
