<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External → POST /auth/exchange | Receives Auth0 token from browser — must verify JWKS signature before trusting claims |
| POST /auth/refresh → Supabase | Refresh token verified before re-signing; expired/tampered tokens rejected |
| GET /auth/me → passport-jwt | Passport validates Bearer token using JwtStrategy before handler executes |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-02-01 | Spoofing | POST /auth/exchange | mitigate | verifyAuth0Token() uses jwks-rsa to fetch Auth0 public key and verify RS256 signature — not skippable |
| T-02-02-02 | Tampering | JWT payload | mitigate | D-06: payload is { sub, email } only — removing orgId/roles prevents privilege escalation via token forgery |
| T-02-02-03 | Information Disclosure | JWT secret leakage | mitigate | JWT_SECRET read from env var in JwtModule config — never in source code |
| T-02-02-04 | Elevation of Privilege | roles hardcode ['owner'] | mitigate | mapToUser() now reads from dbRow.roles — roles set by DB, not hardcoded in code |
| T-02-02-05 | Repudiation | User auto-provisioning | accept | User row created on first exchange; Auth0 sub is the authoritative identity source |
| T-02-02-06 | Denial of Service | JWKS fetch on every exchange | accept | jwks-rsa caches keys with configurable TTL — already configured in existing code |
</threat_model>
