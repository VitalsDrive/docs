<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| FMC003 device → TCP parser | Raw binary frames arrive over untrusted network; CRC is the primary integrity check |
| Simulator → Parser | Internal test traffic; correctness here validates the protocol implementation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-03-01 | Tampering | TCP frame / CRC field | mitigate | CRC-16/IBM (now correctly implemented) rejects any single-bit flip with ~99.998% probability; multi-byte corruption detected probabilistically |
| T-01-03-02 | Tampering | Data Field Length header | mitigate | Parser reads exact declared length and validates CRC over the declared region only — inflated length that was bug CR-02 allowed reading 2 bytes past payload into CRC region, now fixed |
| T-01-03-03 | Spoofing | IMEI auth handshake | accept | Single-byte 0x01 ACK with no challenge; acceptable for internal MVP fleet where IMEI list is operator-controlled. Phase 2 may add device certificates |
| T-01-03-04 | Denial of Service | Parser TCP buffer | accept | Parser accumulates until preamble found and discards at >100 bytes; acceptable for current single-device MVP load |

Note: CR-01 through CR-03 were effectively a CRC bypass — the wrong algorithm meant the
parser accepted packets that a real FMC003 would never emit (matching wrong checksums
between simulator and parser masked the bug). This plan closes the gap.
</threat_model>
